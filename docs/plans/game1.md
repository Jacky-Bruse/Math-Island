# 《机器人快递员：星球配送任务》可实施方案

## 1. 方案结论

在现有“启明星”项目中增加一个面向 5–7 岁儿童的单机路径规划游戏。

MVP 采用以下确定方案：

- 地图由节点和可见航线组成；
- 玩家只能沿航线移动到相邻节点；
- 从机器人起点出发，收集全部能量晶体，每个节点最多经过一次，最后抵达快递终点；
- 首版提供 20 个固定关卡，不做运行时随机生成；
- 使用开发期求解器校验每关存在且仅存在一个解；
- 复用项目现有 React、TypeScript、SVG、CSS 动画、声音设置和本地存储；
- 不新增 Framer Motion、Howler.js 或其他依赖。

这套规则能够形成真实的路径规划，同时把首版开发量和随机生成风险控制在合理范围内。

---

## 2. 游戏目标

### 目标用户

- 5–7 岁儿童；
- 主要使用平板触屏；
- 无需阅读复杂说明即可开始游戏。

### 训练目标

- 观察节点之间的连接关系；
- 提前规划经过顺序；
- 识别走入死路的风险；
- 学习撤销并调整方案。

### 核心体验

> 看清航线，想好顺序，帮助机器人一次收齐能量晶体并完成配送。

---

## 3. MVP 范围

### 必须实现

- 游戏选择页入口；
- 20 个固定关卡；
- SVG 响应式地图；
- 触摸、鼠标 Pointer Events 操作；
- 节点吸附与相邻节点判定；
- 已走路径显示；
- 撤销一步、重新开始、提示；
- 通关判定和简单庆祝动画；
- 1–3 星评价；
- 关卡解锁和最佳星级本地保存；
- 单页关卡选择面板；
- 遵循现有声音开关、深色主题和清空数据功能；
- 关卡合法性自动检查。

### 首版不实现

- 运行时随机关卡；
- 50 关内容；
- 障碍物和连续轨迹碰撞；
- 最短距离、限制时间或限制步数；
- 徽章、皮肤、商城和成长系统；
- 每日挑战、排行榜、云存档和家长统计；
- 强制全屏或强制横屏。

## 4. 核心规则

地图是一个无向图：节点是地点，边是机器人可以通行的航线。

玩家必须满足：

1. 从 `start` 节点开始；
2. 每次只能前往当前节点的相邻节点；
3. 每个节点最多访问一次；
4. 收集全部 `checkpoint`；
5. 最后进入 `target` 节点。

满足全部条件即通关。

### 目标节点规则

- 尚有能量晶体未收集时，进入终点的操作不生效；
- 终点闪烁并提示“还有能量晶体没有收集”；
- 玩家保留当前路径，可以撤销或继续选择其他相邻节点。

### 错误操作规则

以下操作不加入路径，也不会重置关卡：

- 拖向不存在航线的节点；
- 拖回已访问节点；
- 未收集全部晶体时进入终点；
- 松手时没有进入任何节点的吸附范围。

错误通过节点晃动、航线变红和短音效反馈，不使用惩罚弹窗。

### 死路规则

合法移动可能让玩家进入无法完成的状态。游戏不立即判失败，玩家可以：

- 点击“撤销”退回一步；
- 点击“提示”获得下一步建议；
- 点击“重新开始”清空当前路径。

---

## 5. 操作设计

### 基本操作

1. 玩家按住机器人或当前节点；
2. 拖向一个相邻节点，或者直接点击相邻节点；
3. 拖动时显示临时路径；
4. 进入节点吸附范围后高亮节点；
5. 拖动时在目标节点松手提交；点击时直接提交，机器人沿航线移动到新节点。

### 输入实现

- 统一使用 Pointer Events，同时支持触摸和鼠标；
- 游戏地图区域设置 `touch-action: none`，页面其他区域保持正常行为；
- 拖动开始后使用 pointer capture，避免手指移出节点时丢失事件；
- SVG 使用固定 `viewBox` 坐标，实际尺寸随容器缩放；
- 吸附半径使用 SVG 坐标计算，不依赖屏幕像素；
- 节点可视半径和触控命中区域分离，保证儿童容易点中。

### 按钮

- 返回：回到 `/games`；
- 撤销：移除路径最后一个节点，起点时禁用；
- 重新开始：立即重置当前关卡，不弹确认框；
- 提示：高亮一条能够继续完成关卡的航线；
- 下一关：仅在通关面板中出现；
- 关卡编号：点击后打开关卡选择面板。

---

## 6. 提示与评分

### 提示算法

求解器基于玩家当前路径寻找一个可完成的剩余路径：

- 存在可完成路径：高亮下一条正确航线；
- 当前状态已经无解：高亮“撤销”按钮，提示“退一步再试试”；
- 每次点击提示只展示一步，不直接播放完整答案。

每次有效点击提示按钮都计一次，包括提示玩家撤销；提示展示期间暂时禁用按钮，避免重复计数。

### 星级规则

错误尝试和撤销不扣星，避免惩罚探索。

- 3 星：未使用提示完成；
- 2 星：使用 1 次提示完成；
- 1 星：使用 2 次及以上提示完成。

每关保存历史最佳星级。重复挑战只能提高成绩，不会降低已有成绩。

## 7. 关卡设计

### 首版策略

20 关全部作为固定数据随前端发布。固定关卡便于：

- 保证难度按预期递增；
- 让关卡编号和存档稳定；
- 逐关检查儿童体验；
- 避免随机布局重叠、无解或多解。

### 难度分段

#### 第 1–3 关：教学

- 3–4 个节点；
- 第 1 关没有分叉；
- 第 2 关演示必须收集全部晶体；
- 第 3 关出现一个明显分叉，并引导使用撤销；
- 首次出现操作时使用图形提示，不显示长段文字教程。

#### 第 4–8 关：入门

- 4–5 个节点；
- 1–2 个分叉；
- 错误顺序会进入简单死路；
- 正确路线可通过观察直接判断。

#### 第 9–14 关：进阶

- 5–7 个节点；
- 多个局部选择；
- 需要提前考虑终点位置和节点度数；
- 地图仍保持清晰，航线尽量不交叉。

#### 第 15–20 关：挑战

- 6–8 个节点；
- 存在相似但错误的路线；
- 允许少量航线交叉，但交叉点不代表节点；
- 每关仍只有一个完整解。

### 关卡制作约束

- 起点和终点各一个；
- 至少一个能量晶体；
- 节点之间不得视觉重叠；
- 航线不得穿过非端点节点；
- 避免两条航线在小角度下贴得过近；
- 每关必须恰好有一个满足规则的完整路径；
- 正确答案不得依赖看不清的交叉线或触控精度。

## 8. 数据模型

```typescript
type RobotCourierNodeType = 'start' | 'checkpoint' | 'target'

interface RobotCourierNode {
  id: string
  x: number
  y: number
  type: RobotCourierNodeType
}

type RobotCourierEdge = readonly [from: string, to: string]

interface RobotCourierLevel {
  id: number
  nodes: readonly RobotCourierNode[]
  edges: readonly RobotCourierEdge[]
}

interface RobotCourierProgress {
  version: 1
  unlockedLevel: number
  bestStars: Partial<Record<number, 1 | 2 | 3>>
}
```

关卡数据使用 TypeScript 常量，不单独使用 JSON。这样可以直接获得类型检查，也不需要维护额外加载逻辑。

玩家当前路径只保存在页面状态中：

```typescript
type PlayerPath = string[]
```

离开未完成关卡时不保存草稿；关卡较短，重新进入后从起点开始。

当前关卡只保存在页面状态中，首次进入默认打开最高已解锁关卡：

```typescript
const [currentLevel, setCurrentLevel] = useState(progress.unlockedLevel)
```

## 9. 核心逻辑

### 合法移动

`canMove(level, path, nextNodeId)` 依次检查：

1. 节点存在；
2. 当前节点和目标节点之间存在边；
3. 目标节点没有访问过；
4. 如果目标是终点，所有能量晶体已经访问。

### 通关判断

`isComplete(level, path)` 检查：

- 路径从起点开始；
- 路径最后是终点；
- 所有能量晶体均已访问；
- 没有重复节点；
- 每一对连续节点之间都存在边。

### 关卡求解器

使用深度优先搜索枚举合法路径，节点上限为 8，无需引入图库或复杂优化。

求解器承担两个职责：

- 开发时验证每关解的数量等于 1；
- 游戏中根据当前路径计算提示。

开发验证找到两个解后即可提前停止，不必枚举全部答案。

## 10. 页面与视觉布局

### 路由和入口

- 游戏列表：`/games`；
- 机器人快递员：`/games/robot-courier`；
- 在现有 `GameSelectPage` 增加游戏卡片；
- 首页“游戏”描述由“数独”改为“数独 / 机器人快递员”。

首版在同一游戏页面内切换关卡，不增加独立关卡选择路由。

点击顶部关卡编号打开 1–20 关选择面板：

- 默认加载最高已解锁关卡；
- 已解锁关卡可以选择，并显示历史最佳星级；
- 未解锁关卡显示锁定且不可选择；
- 通关后“下一关”直接切换到新解锁关卡；
- 第 20 关完成后再次进入，默认打开第 20 关并显示已全部通关。

### 布局

- 顶部：返回、关卡编号、累计星级；
- 中部：自适应 SVG 地图；
- 底部：撤销、重新开始、提示；
- 通关后：覆盖式轻量面板，显示星级和“下一关”。

### 屏幕方向

现有 PWA manifest 配置为 `orientation: 'portrait'`。实现本游戏时将其改为 `orientation: 'any'`，允许整个应用随设备方向旋转；现有页面继续使用响应式布局：

- 横屏时地图区域更宽，为最佳体验；
- 竖屏时使用紧凑工具栏和较高地图区域，保持可玩；
- 特别窄的屏幕可以显示非阻塞的“横屏体验更好”提示；
- 不调用方向锁定 API；
- 不要求浏览器全屏，PWA standalone 和普通浏览器均可运行。

### 视觉和反馈

- 延续启明星现有圆角、明亮色彩和大触控区域；
- 星空背景使用 CSS 渐变和少量静态装饰，不引入粒子库；
- 机器人、晶体和包裹首版使用简单 SVG 图形或 emoji；
- 已访问晶体降低亮度并播放一次收集动画；
- 当前可移动节点保持柔和高亮；
- 航线交叉处不绘制节点，避免误解；
- 动画遵循 `prefers-reduced-motion`。

## 11. 与现有项目集成

### 复用能力

- 使用现有 `HashRouter` 和 `/games` 层级；
- 使用 `PageContainer`、`BackButton` 和现有主题色变量；
- 通过 `useSettings()` 读取声音开关；
- 复用 `useSound()` 的 `click`、`wrong`、`hint`、`complete`；
- 使用现有 localStorage 封装保存关卡进度；
- 将新存储 key 加入 `clearAllData()`。

### 建议文件

```text
src/
  pages/RobotCourierPage.tsx
  components/robot-courier/RobotCourierBoard.tsx
  lib/robot-courier.ts
  lib/robot-courier-levels.ts
  types/robot-courier.ts
scripts/
  test-robot-courier.ts
```

只在组件确实变得难以阅读时再拆分机器人、节点或路径组件，首版不为每个 SVG 元素创建独立文件。

### 存储 key

```text
math-island:robot-courier-progress
```

读取损坏数据或版本不匹配时恢复默认进度：第 1 关解锁、没有历史星级。

读取时只接受以下有效数据：

- `version === 1`；
- `unlockedLevel` 是 1–20 的整数；
- 关卡 ID 在 1–20 范围内；
- 星级只能是 1、2、3。

无效字段忽略；整体结构无法识别时恢复默认进度。

## 12. 状态流转

```text
进入页面
  → 读取本地进度
  → 加载最高已解锁关卡
  → planning

选择关卡
  → 只允许选择已解锁关卡
  → 清空当前路径并加载目标关卡
  → planning

planning
  → 合法移动：更新路径
  → 非法移动：反馈后保持 planning
  → 撤销/重开：更新或清空路径
  → 满足通关条件：completed

completed
  → 计算星级
  → 保存最佳星级和解锁进度
  → 重玩本关或进入下一关
```

最后一关完成后显示“全部配送完成”，提供“重玩本关”和“返回游戏列表”。

## 13. 验证与验收

### 最小自动检查

`scripts/test-robot-courier.ts` 使用项目已有的 `tsx` 运行，不引入测试框架。`package.json` 增加：

```json
"test:robot-courier": "tsx scripts/test-robot-courier.ts"
```

自检脚本至少验证：

- 20 个关卡 ID 连续且节点引用有效；
- 每关恰好一个起点和一个终点；
- 边的两端均为已定义节点；
- 不允许自环和重复边；
- 每关求解结果恰好为一个；
- 非相邻移动、重复节点和提前进入终点会被拒绝；
- 完整正确路径能够通关；
- 游戏入口和路由存在。

### 手工验收

- 触屏拖动不会滚动游戏地图；
- 鼠标可以完成相同操作；
- 点击相邻节点可以移动；
- 竖屏和横屏都能看到完整控制栏；
- 安装为 PWA 后可以在竖屏和横屏之间旋转；
- 每个节点容易命中，航线和交叉点容易理解；
- 关卡面板只允许进入已解锁关卡，并显示最佳星级；
- 撤销、重开和提示在所有路径状态下行为一致；
- 关闭声音后没有游戏音效；
- 浅色和深色主题均可辨认；
- 刷新页面后关卡解锁和最佳星级保留；
- 清空本地数据后游戏进度恢复默认；
- 离线重新打开已安装 PWA 时可以进入游戏。

## 14. 实施顺序

### 第一阶段：规则闭环

- 定义类型和 3 个示例关卡；
- 实现合法移动、通关判断和 DFS 求解器；
- 完成关卡数据自动检查。

### 第二阶段：可玩页面

- 接入游戏入口和路由；
- 实现 SVG 地图、Pointer Events、节点吸附和路径状态；
- 实现拖动/点击移动、撤销、重开、提示和通关流程；
- 实现单页关卡选择面板。

### 第三阶段：内容和集成

- 补齐并验证 20 个固定关卡；
- 接入星级、解锁、本地存储、主题和声音设置；
- 将 PWA `orientation` 改为 `any`；
- 完成横竖屏适配、儿童反馈和离线验收。

每个阶段都保持可运行；先确认核心玩法好玩，再投入完整关卡和视觉资源。

## 15. 后续扩展条件

只有在 20 个固定关卡验证玩法有效后，再考虑：

- 需要持续新内容时：增加基于种子的运行时关卡生成，并继续用求解器验解；
- 儿童确实需要长期目标时：增加机器人皮肤或徽章；
- 有明确教学评估需求时：增加本地家长统计；
- 固定图玩法已经稳定时：再评估障碍物、单向航线或传送节点。

后续扩展不得改变已有固定关卡和存档含义，进度结构变化时提升 `version` 并提供迁移或安全重置。
